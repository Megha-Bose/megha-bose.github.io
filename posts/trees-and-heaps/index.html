Trees & Heaps - When Flat Structures Hit Their Limit

Aug 3, 2021
16 min read

Prerequisites:

  • Data Structures - Storage Is Strategy

Every file system on your computer is a tree. Every DOM in a web browser - the structure that defines what a webpage looks like - is a tree. The decision trees inside chess engines are trees. Phylogenetic trees in biology show evolutionary relationships. Organizational hierarchies are trees.

Trees are not an abstract data structure invented by computer scientists. They are the shape that hierarchical data naturally takes. When things have parents and children, a tree is what you get.

What Is a Tree?

A tree is a connected graph with no cycles and a designated root node. Every node except the root has exactly one parent. Every node can have zero or more children. Nodes with no children are called leaves. We always draw the root at the top and the leaves at the bottom - the exact opposite of an actual tree. Computer scientists apparently spent too much time indoors.

A binary tree is a tree where every node has at most two children, called left and right.

A few vocabulary items you’ll encounter constantly:

  • Height of a node: the length of the longest path from that node down to a leaf. The height of a leaf is 0. The height of the tree is the height of the root.
  • Depth of a node: the length of the path from the root down to that node. The root has depth 0.
  • Complete binary tree: all levels are fully filled except possibly the last, which is filled from left to right.
  • Full binary tree: every node has either 0 or 2 children (no nodes with exactly 1 child).
  • Perfect binary tree: all interior nodes have 2 children and all leaves are at the same depth. A perfect binary tree of height $h$ has $2^{h+1} - 1$ nodes.
flowchart TD A["root\n(depth 0)"] --> B["(depth 1)"] & C["(depth 1)"] B --> D["leaf\n(depth 2)"] & E["leaf\n(depth 2)"] C --> F["leaf\n(depth 2)"]

A binary tree of height $h$ has at most $2^{h+1} - 1$ nodes. Conversely, a binary tree with $n$ nodes has height at least $\log_2 n$. This logarithmic relationship is the source of $O(\log n)$ performance for tree operations.

Binary Search Trees: O(log n) with a Condition

A binary search tree (BST) adds one invariant to the binary tree structure: for every node, all keys in its left subtree are smaller than the node’s key, and all keys in its right subtree are larger.

flowchart TD A["8"] --> B["3"] & C["12"] B --> D["1"] & E["6"] C --> F["10"] & G["15"]

This invariant makes searching simple. To find key $k$: start at the root. If $k$ equals the root’s key, you’re done. If $k$ is smaller, recurse into the left subtree. If $k$ is larger, recurse into the right subtree. Each comparison eliminates one subtree entirely.

The time is $O(h)$ where $h$ is the height of the tree. For a balanced tree, $h = O(\log n)$, giving $O(\log n)$ search. This is binary search, but on a dynamic structure that supports insertion and deletion efficiently.

Insertion is similar: search for where the key would be, then insert a new leaf there.

Before discussing deletion, it helps to know the three standard ways to visit all nodes of a tree:

  • In-order (left → node → right): visits nodes in sorted order for a BST. If you read off the keys during an in-order traversal of a BST, you get them smallest to largest.
  • Pre-order (node → left → right): visits the node before its subtrees. Used to copy or serialize a tree.
  • Post-order (left → right → node): visits the node after its subtrees. Used to delete a tree or evaluate expression trees.

Deletion is the tricky case. Three scenarios:

  • Node is a leaf: remove it directly.
  • Node has one child: splice it out by linking its parent to its child.
  • Node has two children: find the in-order successor (the smallest key in the right subtree - the leftmost node going right). Copy that key into the current node and delete the in-order successor (which has at most one child, so it’s an easier case).

The problem: the $O(h)$ complexity is only $O(\log n)$ if the tree is balanced. Feed a BST elements in sorted order and it degenerates into a linked list, with every left subtree empty and the tree having height $n - 1$. Searching this takes $O(n)$.

flowchart TD A["1"] --> B["2"] B --> C["3"] C --> D["4"] D --> E["5"]

AVL and Red-Black Trees: Keeping the Promise

The fix for BST degeneration is balancing: enforce that the height stays $O(\log n)$ through rotations that restructure the tree.

AVL trees maintain the invariant that for every node, the heights of its left and right subtrees differ by at most 1. After any insertion or deletion, walk back up the tree checking this invariant. If a node violates it, apply a rotation - a local restructuring operation that rearranges a small subtree without breaking the BST property.

There are four rotation cases: left-left, right-right, left-right, and right-left (the last two are double rotations). To track imbalance, each node stores a balance factor (bf) - the height of its right subtree minus the height of its left subtree. An AVL tree requires every node’s balance factor to be -1, 0, or +1. A value of -2 means the left side is two levels deeper than the right - a violation that triggers a rotation. Here is the simplest case - a right rotation fixing a left-left imbalance, where inserting A caused C’s balance factor to hit -2:

flowchart LR subgraph after["After rotation"] B2["B\nbf = 0"] --> A2["A\nbf = 0"] & C2["C\nbf = 0"] end subgraph before["Before (violation at C)"] C1["C\nbf = -2"] --> B1["B\nbf = -1"] B1 --> A1["A\nbf = 0"] end

The key insight is that a single or double rotation is always enough to fix a violation at one node, and at most $O(\log n)$ nodes need checking after an insertion. This keeps all operations at $O(\log n)$.

Seeing rotations happen interactively makes them much easier to understand than reading about them. Try inserting a few values into this AVL tree visualizer and watch the rotations fire.

Red-black trees use a different invariant: each node is colored red or black, with four rules that together ensure the tree’s height is at most $2 \log_2(n+1)$. The key rules are: no two consecutive red nodes on any path, and every path from root to a null leaf has the same number of black nodes. This is what prevents the tree from becoming lopsided.

flowchart TD A["7"]:::black --> B["3"]:::red & C["18"]:::red B --> D["1"]:::black & E["5"]:::black C --> F["10"]:::black & G["22"]:::black classDef black fill:#222,color:#fff,stroke:#444 classDef red fill:#c0392b,color:#fff,stroke:#c0392b

Red-black trees require fewer rotations per insertion and deletion than AVL trees, making them faster in write-heavy workloads. They’re used in C++’s std::map, Java’s TreeMap, and the Linux kernel’s process scheduler.

You don’t need to memorize the rotation cases. What matters is the guarantee: balanced BSTs give $O(\log n)$ for search, insert, and delete, always, regardless of insertion order.

Heaps: Trees Stored in Arrays

A binary heap is a complete binary tree satisfying the heap property: every parent is greater than or equal to both its children (max-heap), or every parent is less than or equal to both its children (min-heap).

The most practically important fact about heaps is how they’re stored.

Because heaps are complete binary trees, we can store them in an array without any pointers. For a 1-indexed array $A[1..n]$:

  • Node $i$ has its parent at $\lfloor i/2 \rfloor$
  • Node $i$ has its left child at $2i$
  • Node $i$ has its right child at $2i + 1$

The root is at index 1. The last element is at index $n$. No pointer traversal needed - just arithmetic.

flowchart TD A["[1] 16"] --> B["[2] 14"] & C["[3] 10"] B --> D["[4] 8"] & E["[5] 7"] C --> F["[6] 9"] & G["[7] 3"]

Discomfort check. “If a heap is a tree, why represent it as an array?” Because the array representation is dramatically faster in practice. Pointer-based trees require one memory allocation per node, and traversing them causes cache misses (jumping between nodes scattered across the heap). An array-based heap stores all nodes contiguously - the parent-child relationships are computed in one instruction, and the data is all nearby in memory. For the same operations, an array heap can be 3-5 times faster than a pointer-based equivalent.

Heap Operations

Heapify (sift down): given a node that may violate the heap property (but whose children’s subtrees are valid heaps), fix it by swapping it with the larger child and recursing. This takes $O(\log n)$ time - one path from node to leaf.

Insert: place the new element at the end of the array (next available position in the complete binary tree), then sift it upward until the heap property is restored. Each swap moves it one level up. Time: $O(\log n)$.

Extract-max: the maximum is always at the root (index 1). Remove it, move the last element to the root position, then sift it down to restore the heap property. Time: $O(\log n)$.

Build-heap: given an arbitrary array, convert it into a valid heap. The naive approach is to insert each element one by one: $O(n \log n)$. But there’s a better way.

Start at the last internal node (index $\lfloor n/2 \rfloor$) and sift down toward the root, calling heapify at each node. Why does this work in $O(n)$ instead of $O(n \log n)$?

Consider the cost of each heapify call. A node at height $h$ from the bottom needs at most $h$ swaps. In a heap of $n$ nodes, there are roughly $n/2^{h+1}$ nodes at height $h$. The total cost is:

$$\sum_{h=0}^{\lfloor \log n \rfloor} \frac{n}{2^{h+1}} \cdot h = \frac{n}{2} \sum_{h=0}^{\infty} \frac{h}{2^h}$$

The sum $\sum_{h=0}^{\infty} h x^h = x/(1-x)^2$. At $x = 1/2$ this equals 2. So the total cost is $\frac{n}{2} \cdot 2 = n = O(n)$.

Intuitively: most nodes are near the leaves, where heapify is cheap. Only a few nodes near the root need to travel far. The work is concentrated where nodes are scarce, so the total remains linear.

Heapsort

Build-heap converts the array to a max-heap in $O(n)$. Then: repeatedly swap the root (the current maximum) with the last element of the heap, shrink the heap size by 1, and heapify the root. After $n-1$ such extractions, the array is sorted.

Total: $O(n)$ for build-heap plus $n \cdot O(\log n)$ for extractions = $O(n \log n)$, in-place, no auxiliary memory. The sort is not stable (heap operations can swap equal elements past each other), and it’s slower than quicksort in practice due to poor cache behavior - the heap’s jumpy access pattern doesn’t benefit from cache prefetching.

B-Trees

Binary search trees work well when data fits in RAM. But what happens when you have a billion records on disk? Disk reads are roughly 100,000 times slower than RAM reads. Every time you follow a pointer to a new node, you might be paying that cost. A balanced BST with a billion records has height around 30 - meaning 30 disk reads per lookup. That is unacceptably slow.

The fix is to make each node fat. Instead of holding one key and two children, a B-tree node holds up to $2t - 1$ keys and up to $2t$ children, where $t$ is called the minimum degree. You size the node so that it fits exactly in one disk block (typically 4KB or 16KB). With hundreds of keys per node, the tree height collapses - a B-tree over a billion records might have height 3 or 4, meaning 3-4 disk reads per lookup.

Every node except the root has between $t - 1$ and $2t - 1$ keys. The keys within a node are sorted. The children of a node interleave with its keys: the $i$-th child subtree contains all keys between the $(i-1)$-th and $i$-th keys of the node. This is the BST property generalized to many children.

flowchart TD A["10 | 20 | 30"] --> B["3 | 7"] & C["13 | 17"] & D["23 | 27"] & E["35 | 40"]

Search works exactly like BST search, generalized: at each node, binary search the sorted keys to find which child to descend into. Each node visit is one disk read. Time: $O(\log_t n)$ disk reads, or $O(t \log_t n)$ key comparisons.

Insertion always happens at a leaf. Search down to the correct leaf and insert the key. If the leaf has room (fewer than $2t - 1$ keys), you’re done. If it’s full, split it: push the median key up to the parent and divide the node into two nodes of $t - 1$ keys each. If the parent is also full, split it too. Splits can cascade up to the root. When the root splits, the tree grows taller by one level - the only way a B-tree gains height.

B+ trees are a variant used by almost every database. Before explaining the difference, here is how a database index actually works.

Your table is just rows stored on disk in no particular order - a flat file. To answer “find all users where age = 30,” the database would normally read every single row and check the age field. With a million rows, that is a million disk reads.

An index on the age column is a B+ tree built separately, where each key in the tree is an age value, and each key points to a list of disk locations - one for every row with that age value. There could be one such row or ten thousand; the leaf node holds all their pointers. To answer the query, the database walks down the B+ tree - at each node it reads one disk block and follows the right branch - arriving at the leaf node that holds age 30 and the list of pointers to every matching row. Instead of a million reads to find them, it takes three or four to reach the leaf, then one read per matching row.

For a range query like “age BETWEEN 25 AND 35,” the database finds the leaf containing 25, then follows the linked chain of leaf nodes rightward until it passes 35, collecting row pointers along the way. The leaf chain is why B+ trees are used over regular B-trees for this purpose.

B+ trees are the standard index structure. The difference from regular B-trees: internal nodes store only keys (no data), and all actual data lives in the leaf nodes. The leaves are also linked together in a sorted linked list. This makes range queries fast - find the start key, then scan the leaf chain without going back up the tree. B+ trees are what PostgreSQL, MySQL, and SQLite use for their indexes.

class BTreeNode:
    def __init__(self, t, leaf=False):
        self.t = t          # minimum degree
        self.leaf = leaf
        self.keys = []
        self.children = []

class BTree:
    def __init__(self, t):
        self.t = t
        self.root = BTreeNode(t, leaf=True)

    def search(self, k, node=None):
        if node is None:
            node = self.root
        i = 0
        while i < len(node.keys) and k > node.keys[i]:
            i += 1
        if i < len(node.keys) and k == node.keys[i]:
            return True
        if node.leaf:
            return False
        return self.search(k, node.children[i])

    def insert(self, k):
        root = self.root
        if len(root.keys) == 2 * self.t - 1:
            # root is full - split it and grow the tree
            new_root = BTreeNode(self.t)
            new_root.children.append(self.root)
            self._split_child(new_root, 0)
            self.root = new_root
        self._insert_non_full(self.root, k)

    def _insert_non_full(self, node, k):
        i = len(node.keys) - 1
        if node.leaf:
            node.keys.append(None)
            while i >= 0 and k < node.keys[i]:
                node.keys[i + 1] = node.keys[i]
                i -= 1
            node.keys[i + 1] = k
        else:
            while i >= 0 and k < node.keys[i]:
                i -= 1
            i += 1
            if len(node.children[i].keys) == 2 * self.t - 1:
                self._split_child(node, i)
                if k > node.keys[i]:
                    i += 1
            self._insert_non_full(node.children[i], k)

    def _split_child(self, parent, i):
        t = self.t
        full = parent.children[i]
        new_node = BTreeNode(t, leaf=full.leaf)
        mid = t - 1
        # push median key up to parent
        parent.keys.insert(i, full.keys[mid])
        parent.children.insert(i + 1, new_node)
        # divide keys and children
        new_node.keys = full.keys[mid + 1:]
        full.keys = full.keys[:mid]
        if not full.leaf:
            new_node.children = full.children[t:]
            full.children = full.children[:t]

Tries

A trie (from “retrieval”, pronounced either “try” or “tree”) is a tree built for string lookups. The key idea: instead of storing whole strings at nodes and comparing them, you store one character per edge. Every path from the root to a marked node spells out a stored string.

This gives you $O(L)$ lookup for a string of length $L$, completely independent of how many strings are in the trie. A hash table also gives $O(L)$ average lookup (since hashing the string takes $O(L)$ time), but a trie also gives you things a hash table cannot: find all strings with a given prefix, find the longest prefix match, enumerate strings in lexicographic order.

Here is a trie storing “cat”, “car”, “can”, and “dog”:

flowchart TD R["root"] --> C["c"] & D["d"] C --> CA["a"] CA --> CAT["t ✓"] & CAR["r ✓"] & CAN["n ✓"] D --> DO["o"] DO --> DOG["g ✓"]

Notice that “cat”, “car”, and “can” all share the prefix “ca” - they follow the same path for the first two characters and only diverge at the third. This shared prefix storage is what makes tries memory-efficient for large collections of strings with common prefixes (like all URLs on a website, or all words in a dictionary).

Each node stores an array (or dictionary) of children indexed by character, and a flag indicating whether that node marks the end of a valid word.

class TrieNode:
    def __init__(self):
        self.children = {}
        self.is_end = False

class Trie:
    def __init__(self):
        self.root = TrieNode()

    def insert(self, word):
        node = self.root
        for ch in word:
            if ch not in node.children:
                node.children[ch] = TrieNode()
            node = node.children[ch]
        node.is_end = True

    def search(self, word):
        node = self.root
        for ch in word:
            if ch not in node.children:
                return False
            node = node.children[ch]
        return node.is_end

    def starts_with(self, prefix):
        """Returns True if any stored word starts with this prefix."""
        node = self.root
        for ch in prefix:
            if ch not in node.children:
                return False
            node = node.children[ch]
        return True

    def all_with_prefix(self, prefix):
        """Returns all stored words that start with the given prefix."""
        node = self.root
        for ch in prefix:
            if ch not in node.children:
                return []
            node = node.children[ch]
        results = []
        self._collect(node, prefix, results)
        return results

    def _collect(self, node, current, results):
        if node.is_end:
            results.append(current)
        for ch, child in node.children.items():
            self._collect(child, current + ch, results)

The all_with_prefix method is what powers autocomplete: given the prefix the user has typed so far, return every word in the dictionary that starts with it. This is $O(L + k)$ where $L$ is the prefix length and $k$ is the number of results returned.
