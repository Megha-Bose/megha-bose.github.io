Trees & Heaps

Aug 3, 2021
8 min read

Prerequisites:

  • Data Structures - Storage Is Strategy

Every file system on your computer is a tree. Every DOM in a web browser - the structure that defines what a webpage looks like - is a tree. The decision trees inside chess engines are trees. Phylogenetic trees in biology show evolutionary relationships. Organizational hierarchies are trees.

Trees are not an abstract data structure invented by computer scientists. They are the shape that hierarchical data naturally takes. When things have parents and children, a tree is what you get.

What Is a Tree?

A tree is a connected graph with no cycles and a designated root node. Every node except the root has exactly one parent. Every node can have zero or more children. Nodes with no children are called leaves. We always draw the root at the top and the leaves at the bottom - the exact opposite of an actual tree. Computer scientists apparently spent too much time indoors.

A binary tree is a tree where every node has at most two children, called left and right.

A few vocabulary items you’ll encounter constantly:

  • Height of a node: the length of the longest path from that node down to a leaf. The height of a leaf is 0. The height of the tree is the height of the root.
  • Depth of a node: the length of the path from the root down to that node. The root has depth 0.
  • Complete binary tree: all levels are fully filled except possibly the last, which is filled from left to right.
  • Full binary tree: every node has either 0 or 2 children (no nodes with exactly 1 child).
  • Perfect binary tree: all interior nodes have 2 children and all leaves are at the same depth. A perfect binary tree of height $h$ has $2^{h+1} - 1$ nodes.

A binary tree of height $h$ has at most $2^{h+1} - 1$ nodes. Conversely, a binary tree with $n$ nodes has height at least $\log_2 n$. This logarithmic relationship is the source of $O(\log n)$ performance for tree operations.

Binary Search Trees: O(log n) with a Condition

A binary search tree (BST) adds one invariant to the binary tree structure: for every node, all keys in its left subtree are smaller than the node’s key, and all keys in its right subtree are larger.

This invariant makes searching simple. To find key $k$: start at the root. If $k$ equals the root’s key, you’re done. If $k$ is smaller, recurse into the left subtree. If $k$ is larger, recurse into the right subtree. Each comparison eliminates one subtree entirely.

The time is $O(h)$ where $h$ is the height of the tree. For a balanced tree, $h = O(\log n)$, giving $O(\log n)$ search. This is binary search, but on a dynamic structure that supports insertion and deletion efficiently.

Insertion is similar: search for where the key would be, then insert a new leaf there. Deletion is the tricky case. Three scenarios:

  • Node is a leaf: remove it directly.
  • Node has one child: splice it out by linking its parent to its child.
  • Node has two children: find the in-order successor (the smallest key in the right subtree - the leftmost node going right). Copy that key into the current node and delete the in-order successor (which has at most one child, so it’s an easier case).

The problem: the $O(h)$ complexity is only $O(\log n)$ if the tree is balanced. Feed a BST elements in sorted order and it degenerates into a linked list, with every left subtree empty and the tree having height $n - 1$. Searching this takes $O(n)$.

AVL and Red-Black Trees: Keeping the Promise

The fix for BST degeneration is balancing: enforce that the height stays $O(\log n)$ through rotations that restructure the tree.

AVL trees maintain the invariant that for every node, the heights of its left and right subtrees differ by at most 1. After any insertion or deletion, walk back up the tree checking this invariant. If a node violates it, apply a rotation - a local restructuring operation that rearranges a small subtree without breaking the BST property.

There are four rotation cases: left-left, right-right, left-right, and right-left (the last two are double rotations). The details of which rotation to apply in which case are mechanical; the key insight is that a single or double rotation is always enough to fix a violation at one node, and at most $O(\log n)$ nodes need checking after an insertion. This keeps all operations at $O(\log n)$.

Red-black trees use a different invariant: each node is colored red or black, and the coloring satisfies four rules that together ensure the tree’s height is at most $2 \log_2(n+1)$. Red-black trees require fewer rotations per insertion and deletion than AVL trees, making them faster in write-heavy workloads. They’re used in C++’s std::map, Java’s TreeMap, and the Linux kernel’s process scheduler.

You don’t need to memorize the rotation cases. What matters is the guarantee: balanced BSTs give $O(\log n)$ for search, insert, and delete, always, regardless of insertion order.

Heaps: Trees Stored in Arrays

A binary heap is a complete binary tree satisfying the heap property: every parent is greater than or equal to both its children (max-heap), or every parent is less than or equal to both its children (min-heap).

The most practically important fact about heaps is how they’re stored.

Because heaps are complete binary trees, we can store them in an array without any pointers. For a 1-indexed array $A[1..n]$:

  • Node $i$ has its parent at $\lfloor i/2 \rfloor$
  • Node $i$ has its left child at $2i$
  • Node $i$ has its right child at $2i + 1$

The root is at index 1. The last element is at index $n$. No pointer traversal needed - just arithmetic.

Discomfort check. “If a heap is a tree, why represent it as an array?” Because the array representation is dramatically faster in practice. Pointer-based trees require one memory allocation per node, and traversing them causes cache misses (jumping between nodes scattered across the heap). An array-based heap stores all nodes contiguously - the parent-child relationships are computed in one instruction, and the data is all nearby in memory. For the same operations, an array heap can be 3-5 times faster than a pointer-based equivalent.

Heap Operations

Heapify (sift down): given a node that may violate the heap property (but whose children’s subtrees are valid heaps), fix it by swapping it with the larger child and recursing. This takes $O(\log n)$ time - one path from node to leaf.

Insert: place the new element at the end of the array (next available position in the complete binary tree), then sift it upward until the heap property is restored. Each swap moves it one level up. Time: $O(\log n)$.

Extract-max: the maximum is always at the root (index 1). Remove it, move the last element to the root position, then sift it down to restore the heap property. Time: $O(\log n)$.

Build-heap: given an arbitrary array, convert it into a valid heap. The naive approach is to insert each element one by one: $O(n \log n)$. But there’s a better way.

Start at the last internal node (index $\lfloor n/2 \rfloor$) and sift down toward the root, calling heapify at each node. Why does this work in $O(n)$ instead of $O(n \log n)$?

Consider the cost of each heapify call. A node at height $h$ from the bottom needs at most $h$ swaps. In a heap of $n$ nodes, there are roughly $n/2^{h+1}$ nodes at height $h$. The total cost is:

$$\sum_{h=0}^{\lfloor \log n \rfloor} \frac{n}{2^{h+1}} \cdot h = \frac{n}{2} \sum_{h=0}^{\infty} \frac{h}{2^h}$$

The sum $\sum_{h=0}^{\infty} h x^h = x/(1-x)^2$. At $x = 1/2$ this equals 2. So the total cost is $\frac{n}{2} \cdot 2 = n = O(n)$.

Intuitively: most nodes are near the leaves, where heapify is cheap. Only a few nodes near the root need to travel far. The work is concentrated where nodes are scarce, so the total remains linear.

Heapsort

Build-heap converts the array to a max-heap in $O(n)$. Then: repeatedly swap the root (the current maximum) with the last element of the heap, shrink the heap size by 1, and heapify the root. After $n-1$ such extractions, the array is sorted.

Total: $O(n)$ for build-heap plus $n \cdot O(\log n)$ for extractions = $O(n \log n)$, in-place, no auxiliary memory. The sort is not stable (heap operations can swap equal elements past each other), and it’s slower than quicksort in practice due to poor cache behavior - the heap’s jumpy access pattern doesn’t benefit from cache prefetching.

B-Trees and Tries: A Brief Survey

Not every tree application uses binary trees.

B-trees are balanced search trees where each node can have many children (hundreds, not just two). They’re designed for disk storage: when data doesn’t fit in RAM, you want to minimize the number of disk reads. A B-tree node is sized to fit exactly one disk block. With hundreds of keys per node, the tree height is tiny - a B-tree over a billion records might have height 3 or 4. Every relational database uses B-trees (or the related B+-tree) for indexes.

Tries (pronounced “tries” or “trees”) are trees optimized for string lookups. Instead of comparing whole keys at each node, a trie branches on individual characters. To look up a string of length $L$, you follow $L$ edges from the root - $O(L)$, independent of how many strings are stored. They’re used for autocomplete, spell-checkers, and IP routing tables.
