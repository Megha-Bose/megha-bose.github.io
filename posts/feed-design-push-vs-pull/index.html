Feed Design: Push vs Pull

Sep 15, 2024
9 min read

Prerequisite: The Hot Partition Problem

Katy Perry posts a tweet. She has 100 million followers. In what sense should all 100 million people see it in their feeds within seconds? And more importantly - should the system even try to make that happen?

This question turns out to be the central design decision in social feed architecture. The answer shapes the entire system: storage requirements, write amplification, read latency, consistency guarantees, and how your infrastructure behaves when a celebrity does something newsworthy at 8 PM on a Tuesday.

The Geometry of the Problem

Consider two variables: the number of people a user follows, and the number of followers a user has.

For a typical user following 500 accounts: loading their feed naively means querying 500 separate data locations, fetching recent posts from each, and merging the results by timestamp. Five hundred round trips (or one large scatter-gather query) per feed load, multiplied by millions of concurrent feed loads, is not a system that survives contact with real traffic.

For a celebrity with 100 million followers posting once: if the system writes that post into each follower’s precomputed feed at post time, that’s 100 million writes triggered by a single action. At one microsecond per write (an optimistic estimate), that’s 100 seconds of serial writes. In a parallel pipeline, it’s still an enormous fan-out operation that consumes significant infrastructure for a single event.

These two problems define the design space. You can solve the read problem by precomputing feeds at write time. You can solve the write problem by computing feeds at read time. You cannot fully solve both simultaneously - the tradeoff is structural.

Push: Fan-Out on Write

The push model precomputes a user’s feed when content is created. When a user posts, the system immediately writes a reference to that post into each follower’s feed. By the time a follower loads their feed, the work is already done - it’s a single key lookup.

The natural data structure for a precomputed feed is a sorted list keyed by timestamp. In Redis, a sorted set per user (ZADD feed:{user_id} {timestamp} {post_id}) works well: timestamp is the score, post ID is the member. Range queries by recency are ZREVRANGEBYSCORE. In Cassandra, a table with (user_id, post_timestamp DESC) as the compound primary key supports efficient time-range scans. The reads are simple and fast: one key, one lookup, return the top N items.

When push works well: follower counts are in the thousands to low hundreds of thousands, posts are infrequent relative to reads, and the user base is active (precomputed feeds are actually read, not just stored and forgotten).

Where push fails:

Write amplification is the fundamental problem. A user with 10 million followers generates 10 million writes per post. Storage cost is $O(\text{followers} \times \text{posts})$ - every post is duplicated across every follower’s feed. Deleting a post means removing it from potentially millions of feed entries, which is expensive and slow. A user unfollowing someone requires cleaning up past feed entries retroactively - or accepting that stale entries will be filtered at read time.

The inactive user problem is subtle but real. Users who haven’t logged in for six months still have feeds being updated. Storage accumulates for feeds no one will read. At scale - Instagram has over 2 billion monthly active users, but many of those are not daily active - the wasted writes and storage for inactive users is significant.

Pull: Fan-Out on Read

The pull model computes a user’s feed on demand. When a user loads their feed, the system queries the post store for recent posts from each account they follow, then merges and sorts the results in application memory.

When pull works well: users follow few accounts, the write path must be simple, or the content changes frequently enough that precomputing is wasteful.

Where pull fails:

Read latency grows with the number of followed accounts. A user following 2,000 accounts requires touching 2,000 data locations per feed load - that’s either 2,000 sequential round trips or one large scatter-gather query. For most social platforms where median follow counts are in the hundreds, this is a difficult path to make fast at scale.

Real-time ranking is expensive. A ranked feed (Instagram’s non-chronological ordering, Twitter’s “you might have missed these”) requires running ML inference at read time against a fresh set of candidates. At feed load time, with millions of concurrent users, this is computationally intensive. You need to fetch more candidates than you’ll show, score them, and return the top N.

Twitter’s Architecture Evolution

Twitter started with a push-based model. Fan-out workers consumed a queue of post events and wrote each new tweet into every follower’s precomputed timeline, stored in a Redis cluster. For most users this worked: reads were fast, timelines were fresh, the system was conceptually simple.

Then Beyoncé announced her pregnancy at the 2013 Grammy Awards.

The fan-out workers were designed to handle ordinary write amplification. A user with 100,000 followers generating a tweet was a significant write operation - 100,000 Redis writes - but manageable. Beyoncé had 58 million followers at the time, and the post happened in real-time during a live broadcast, meaning millions of users were actively refreshing Twitter simultaneously.

The fan-out queue grew faster than the workers could drain it. Timelines fell behind. Users saw stale feeds. The system didn’t crash, but it visibly lagged - the timeline gap became a running joke on the platform.

The solution Twitter eventually shipped is the hybrid model, which is now the standard architecture across large social platforms.

The Hybrid Model

The hybrid model draws a line at a follower threshold. Accounts below the threshold (typical users) use push fan-out on write. Accounts above the threshold (celebrities, major media accounts) are excluded from the push fan-out.

When a user loads their feed, the system:

  1. Loads the user’s precomputed feed (populated via push fan-out from non-celebrity accounts they follow)
  2. For each celebrity account the user follows, issues a pull query against the celebrity’s post store
  3. Merges the two streams, sorted by timestamp (or score, for a ranked feed)

This bounds both failure modes. Write amplification is bounded: no fan-out for accounts above the threshold, regardless of follower count. Read latency is bounded: pull only applies to a small number of celebrity accounts, not to all followed accounts.

The system must classify accounts dynamically. A user who gains a million followers becomes a celebrity; a celebrity who loses followers might become a regular user. The threshold crossing requires operational decisions: do existing followers' precomputed feeds get backfilled with the newly-reclassified account’s recent posts?

Instagram uses a similar model. Their engineering blog describes the split point at roughly 100,000 followers, with real-time adjustment based on actual traffic patterns.

Feed Storage: Redis Sorted Sets and Cassandra

Redis sorted sets are the canonical choice for hot, recently-active user timelines. The data structure is exactly right: scores (timestamps) enable efficient range queries, insertion is O(log N), and retrieval of the top K items is O(log N + K). Most platforms cap feed depth (keeping only the most recent 1,000 posts per user) to bound memory usage: ZREMRANGEBYRANK trims the set after each insertion.

For cold users - inactive for days or weeks - feeds are evicted from Redis. On next login, the feed is rebuilt from the Cassandra backing store. This cold-start rebuild is the cost of keeping Redis memory usage bounded.

Cassandra serves as the durable, long-term store. A table with partition key user_id and clustering key post_timestamp DESC supports efficient time-range queries. Cassandra’s wide rows handle large fan-outs naturally - a user’s entire timeline fits in a single partition. For the fan-out operation, Cassandra batches help, though they must be used carefully to avoid overloading partition coordinators.

Kafka as the Fan-Out Backbone

The fan-out operation itself - notifying each follower’s feed when a user posts - is naturally event-driven. When user A posts, a post event is published to a Kafka topic. A fleet of fan-out workers consumes from this topic and writes to each follower’s Redis sorted set (for active users) and Cassandra row (for durability).

Kafka provides the durability and backpressure that a push model needs. If fan-out workers fall behind (during a celebrity post spike), the event stays in Kafka until a worker can process it. The fan-out lag is visible (Kafka consumer lag metrics) and bounded. The post itself is written immediately to the author’s post store; the fan-out is best-effort asynchronous.

ML-Ranked Feeds

Chronological feeds are straightforward to build. Ranked feeds - Instagram’s current default, Twitter’s algorithmic feed, Facebook’s News Feed - are a fundamentally different beast.

Ranking requires, at minimum: candidate retrieval (fetch the last N posts from followed accounts), feature computation (engagement signals, recency, relationship strength, post-type preferences), and scoring (running candidates through a ranking model). This pipeline must complete within feed load latency constraints - typically under 200ms.

Most platforms solve this by pre-computing ranked candidates periodically (every few minutes) and caching the ranked feed. The feed you see may be a few minutes stale, but the ranking computation didn’t happen in your request’s critical path. When you scroll past the precomputed candidates, the system fetches and ranks more - this is why “infinite scroll” loads the first page instantly and occasionally shows a spinner.

Pagination: Why Offset-Based Is Wrong

Offset-based pagination (LIMIT 20 OFFSET 400) seems obvious and is wrong. Computing offset 400 requires skipping 400 rows - expensive at the database level. Worse: if new posts arrive while you’re paginating, all offsets shift. Page 2 may contain items you already saw on page 1, or skip items entirely.

Cursor-based pagination uses the last-seen post’s timestamp (or ID) as an opaque cursor. The next-page query is WHERE timestamp < cursor_timestamp LIMIT 20. This is an O(1) index lookup, is stable across new arrivals, and works identically in Redis (ZREVRANGEBYSCORE with the cursor as the max score) and Cassandra (WHERE post_timestamp < cursor LIMIT 20).

The Decentralized Future

Bluesky’s AT Protocol and Mastodon’s ActivityPub represent a different model: decentralized feeds where the feed algorithm is itself replaceable. In AT Protocol, feeds are computed by external “feed generators” - third-party services that define their own ranking and selection logic. A user can choose their feed algorithm the same way they choose their podcast app.

This shifts the architectural problem: instead of one company’s infrastructure handling all fan-out, the protocol standardizes how posts are published and discovered, and the computation is distributed. The hot partition problem doesn’t go away - it just changes who owns it.

ML-ranked feeds are also under pressure. The recommendation that maximizes engagement can systematically surface outrage and division - this is a well-documented effect. Chronological and user-controlled algorithmic feeds are being re-explored not just for technical reasons but for product and societal ones.

Summary

Model Read Cost Write Cost Storage Consistency Best For
Push (fan-out on write) O(1) O(followers) High (duplicated) Strong Low-follower-count users
Pull (fan-out on read) O(following) O(1) Low (single copy) Strong Low-following-count users
Hybrid O(celebrity count) O(non-celebrity followers) Medium Strong Large-scale social platforms
