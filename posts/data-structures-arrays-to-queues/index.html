Data Structures: Arrays to Queues

Sep 15, 2024
11 min read

Prerequisite: Python from Scratch

Picture two ways to store a list of a thousand names. In the first, you write every name in a notebook, one after another on consecutive lines. In the second, you write each name on a separate index card, and attach a small sticky note to each card pointing to where the next one is. Both hold a thousand names. But if I ask you to hand me the 500th name, the notebook gets you there in a second - count to line 500. The index cards? You have to follow the chain, card by card, all the way through.

That intuition, in compressed form, explains why data structure choice matters enormously, why arrays and linked lists exist as distinct things, and why the answer is not always “use whatever is theoretically faster.” The gap between theory and practice is where cache lines live, and cache lines change everything.

Why We Have Different Structures at All

Every data structure is a negotiation between opposing forces. You want fast access? You pay with slow insertion. You want fast insertion anywhere? You pay with slow access. You want to always retrieve the smallest element instantly? You accept that arbitrary lookups cost more. There is no structure that wins on every axis simultaneously - if there were, we would only have one.

The two axes we care about most are time (measured in Big-O terms) and memory (how much space, and how is it laid out). Big-O notation captures how cost scales with input size: $O(1)$ means constant time regardless of size, $O(n)$ means it grows linearly, $O(\log n)$ means it grows logarithmically (much slower than linear). A million-item structure with $O(1)$ access and a ten-item structure with $O(1)$ access take the same time; that is the promise of constant-time operations.

But there is a third axis that Big-O notation deliberately ignores: cache behavior. Modern CPUs are thousands of times faster than RAM. To compensate, they keep small amounts of recently-used data in caches (L1, L2, L3) that sit physically close to the processor. When you access memory that is adjacent to memory you just accessed, there is a good chance it is already in cache - a cache hit, essentially free. When you jump to a random distant memory location, you might wait 200+ processor cycles for RAM to respond - a cache miss, brutally expensive.

This is why the choice between arrays and linked lists is not the theoretical tie that textbooks sometimes suggest.

Arrays: The Power of Contiguous Memory

An array stores its elements in a single contiguous block of memory. Element 0 sits at some address. Element 1 sits immediately after. Element 42 sits exactly 42 element-widths away from the start. The CPU can compute the address of any element directly:

address(i) = base_address + i * element_size

This is why array access is $O(1)$. It is not “fast because computers are fast.” It is fast because it is a single arithmetic operation.

Python lists are backed by a dynamic array. They are not statically sized - they grow. Here is how that works:

fruits = ["apple", "banana", "cherry"]

# O(1) random access - computes address directly
print(fruits[1])   # "banana"

# O(1) amortized append
fruits.append("date")

# O(n) insert at front - every element shifts right
fruits.insert(0, "avocado")

The “amortized $O(1)$” for append deserves explanation, because the raw truth is messier. When a Python list runs out of space, it allocates a new, larger block (roughly 1.125x to 2x the size) and copies everything over - an $O(n)$ operation. But because this happens exponentially less often as the list grows, the average cost per append, amortized over many appends, approaches $O(1)$. You pay occasionally for a big batch of work, but averaged over time, it is cheap. This is the same logic as a bank account: a single large deposit doesn’t mean every transaction is expensive.

What arrays are good at: anything where you know the index of the thing you want, or where you access elements in order (sequential access plays beautifully into cache prefetching - CPUs can predict you’ll want the next element and load it before you ask).

What arrays are bad at: inserting or deleting in the middle, which requires shifting every subsequent element.

Linked Lists: Flexibility at a Cost

A linked list stores elements in nodes scattered across memory. Each node holds its value and a pointer to the next node. The list itself just holds a pointer to the first node. There is no arithmetic you can do to find the nth element - you must walk the chain.

class Node:
    def __init__(self, val):
        self.val = val
        self.next = None

class LinkedList:
    def __init__(self):
        self.head = None

    def prepend(self, val):
        node = Node(val)
        node.next = self.head
        self.head = node   # O(1): just update the head pointer

The theoretical advantages are real: $O(1)$ insert or delete at the head (update a pointer, done), no need to shift elements, and no fixed-size allocation. But the practical story is grim for most use cases.

The cache miss problem. Each node is allocated separately, meaning they sit at arbitrary locations in memory. Walking a linked list means jumping to a new, probably uncached memory location for every single node. Each jump can cost 200+ CPU cycles waiting for RAM. An array traversal, by contrast, is mostly cache hits - sequential memory that the CPU prefetcher loads ahead of you. In benchmarks on modern hardware, iterating over a million-element array is often 5-10x faster than iterating over a million-element linked list, even though both are $O(n)$. The constant factor hidden inside Big-O is not hiding a small difference here; it is hiding an order-of-magnitude difference.

Python makes it worse. In Python, linked list nodes also carry object overhead (reference counts, type pointers, garbage collector fields). A Python integer stored in a node is not 8 bytes - it is closer to 28 bytes, plus the pointer to the next node, plus the node object itself. The memory footprint is large and fragmented. CPython’s garbage collector adds further pressure when many small objects are alive simultaneously.

The honest conclusion: linked lists in Python are rarely the right answer. For the use case where you genuinely need $O(1)$ front insertion and deletion, reach for collections.deque, which is implemented in C as a doubly-linked list of fixed-size memory blocks - it gets most of the flexibility without the per-node cache-miss penalty.

Stacks: LIFO and the Call Stack

A stack is an access discipline, not a new memory structure. It enforces Last In, First Out (LIFO): you can only add or remove from the top. The canonical metaphor is a stack of plates - you can only take from the top, and you can only add to the top.

Python lists work perfectly as stacks because .append() and .pop() are both $O(1)$:

stack = []
stack.append("first")
stack.append("second")
stack.append("third")

top = stack.pop()   # "third"
print(top)
print(stack)        # ["first", "second"]

The most important stack in your computer is the call stack. Every time you call a function, the CPU pushes a new frame onto the call stack: the function’s local variables, its return address (where to jump back to when it returns), and some bookkeeping. When the function returns, its frame is popped. This is why infinite recursion raises RecursionError - it overflows the stack.

def factorial(n):
    if n <= 1:
        return 1
    return n * factorial(n - 1)   # each call pushes a frame

The OS allocates a fixed-size stack (typically 1-8 MB per thread). Exceed it, and you crash. This is not a Python limitation - it is a fundamental feature of how function calls work at the hardware level.

Other places stacks appear in production systems: undo/redo functionality (each action is pushed; undo pops), expression evaluation (compilers use stacks to handle operator precedence), browser history (back button is a stack pop), and the DFS traversal of graphs.

Queues: FIFO and Event Loops

A queue enforces First In, First Out (FIFO): the first thing added is the first thing removed. Think of a line at a coffee shop, or the message queue between two services.

The critical mistake is using a plain Python list as a queue:

# DO NOT DO THIS
queue = []
queue.append("first")   # O(1), fine
queue.pop(0)            # O(n) - shifts every element left, terrible

list.pop(0) is $O(n)$ because Python must shift every remaining element one position to the left. For a large queue, this is a serious performance problem. Use collections.deque:

from collections import deque

queue = deque()
queue.append("first")
queue.append("second")
queue.append("third")

front = queue.popleft()   # O(1) - "first"

deque (double-ended queue) supports $O(1)$ appends and pops from both ends. It is the right tool for queues, sliding window problems, and BFS traversal.

Where queues appear in real systems: event loops (Node.js and Python’s asyncio both process events from a queue), message brokers (Kafka, RabbitMQ, SQS - each is a persistent, distributed queue), print spoolers, and breadth-first search.

from collections import deque

def bfs(graph: dict, start: str) -> list:
    visited = set()
    queue = deque([start])
    order = []

    while queue:
        node = queue.popleft()
        if node in visited:
            continue
        visited.add(node)
        order.append(node)
        for neighbor in graph.get(node, []):
            queue.append(neighbor)

    return order

graph = {"A": ["B", "C"], "B": ["D"], "C": ["D"], "D": []}
print(bfs(graph, "A"))   # ['A', 'B', 'C', 'D']

The queue is what makes BFS traverse level-by-level rather than depth-first. The structure determines the algorithm’s behavior.

Deques: The Best of Both Worlds

collections.deque is worth dwelling on as its own structure. It is not just “a faster queue” - it is a genuinely useful tool for any problem where you need efficient access to both ends.

The sliding window maximum problem is a classic example: given an array and a window size $k$, find the maximum in each window as it slides across. A naive approach is $O(nk)$; a deque-based approach is $O(n)$ because you maintain a deque of indices where the front always holds the index of the current maximum:

from collections import deque

def sliding_window_max(nums: list[int], k: int) -> list[int]:
    dq = deque()   # stores indices
    result = []

    for i, num in enumerate(nums):
        # remove indices outside the current window
        while dq and dq[0] < i - k + 1:
            dq.popleft()
        # remove indices whose values are smaller than current (they can't be max)
        while dq and nums[dq[-1]] < num:
            dq.pop()
        dq.append(i)
        if i >= k - 1:
            result.append(nums[dq[0]])

    return result

This only works because deque gives $O(1)$ operations on both ends. A list would make the front operations $O(n)$, destroying the whole benefit.

The Cache Locality Story: Arrays Win in Practice

Let me be direct about a conclusion that surprises many newcomers: despite linked lists' theoretical advantages in insertion and deletion, arrays dominate in practice for almost all workloads on modern hardware. The reason is cache locality.

When you access array[i], the CPU loads not just that element but an entire cache line - typically 64 bytes, or 8 to 16 adjacent elements. If you then access array[i+1], it is probably already in cache. The CPU’s hardware prefetcher also predicts sequential access patterns and loads ahead. This means array traversal is extraordinarily cache-friendly.

Linked list traversal, by contrast, follows a pointer to a node that could be anywhere in the virtual address space. Each node access is likely a cache miss. The effect compounds: even if individual operations are $O(1)$ in theory, the wall-clock time for iterating through a large linked list is much higher than iterating through an array, because you are paying hundreds of nanoseconds per cache miss instead of a few nanoseconds for a cache hit.

This is why modern systems programming languages (Rust, C++, Go) push developers toward Vec/array-backed structures, and why Python’s CPython implementation puts deque in a much smaller number of use cases than textbooks would suggest.

Cache-oblivious data structures are an active research area trying to design structures that perform well regardless of cache hierarchy without needing to know cache sizes. Van Emde Boas trees and cache-oblivious B-trees are examples. CPU prefetching is also getting smarter - hardware in modern chips can handle stride-1 and stride-2 patterns automatically. The direction of travel favors arrays even more as CPUs evolve.

The Balanced Parentheses Classic

Here is the stack application that appears in almost every interview and every compiler:

def is_balanced(s: str) -> bool:
    stack = []
    pairs = {")": "(", "]": "[", "}": "{"}

    for char in s:
        if char in "([{":
            stack.append(char)
        elif char in ")]}":
            if not stack or stack[-1] != pairs[char]:
                return False
            stack.pop()

    return len(stack) == 0

print(is_balanced("({[]})"))   # True
print(is_balanced("({[})"))    # False

The insight is that the most recently opened bracket must be closed first - a perfect LIFO problem. Compilers use the same principle to validate syntax.

Summary

Structure Access Insert Front Insert End Notes
list (array) $O(1)$ $O(n)$ $O(1)$ amortized Best cache behavior; use by default
deque $O(n)$ $O(1)$ $O(1)$ Use for queues and sliding windows
Stack (via list) $O(1)$ top only - $O(1)$ push LIFO; call stack, undo, DFS
Queue (via deque) $O(1)$ front only - $O(1)$ enqueue FIFO; BFS, event loops, message queues
Linked list $O(n)$ $O(1)$ $O(1)$ Rarely worth it in Python; cache-hostile

The practical takeaway: use list for almost everything, deque when you need efficient front operations, and be skeptical of linked lists in Python - the cache-miss cost is real and often dominates.
