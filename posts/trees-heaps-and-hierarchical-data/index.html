Trees, Heaps & Hierarchical Data - Structure That Makes Retrieval Fast

Sep 15, 2024
13 min read

Helpful context: Data Structures: Arrays to Queues

Before you ever heard the word “algorithm,” you were already fluent in hierarchical structure. The folder on your desktop contains subfolders, which contain files. Your company has a CEO who reports to a board, with VPs below, then directors, then managers, then engineers. The HTML of the page you are reading right now is a tree: one root element containing children, each of which may have children of their own.

Trees are not just another data structure. They are the natural way to represent things that contain other things. And once you understand them, you start seeing them everywhere: in how expressions are evaluated, how databases are indexed, how operating systems schedule processes, how Dijkstra’s algorithm finds shortest paths.

This post focuses on three tree variants that appear constantly in real systems - binary search trees (BSTs), heaps, and the conceptual foundation for B-trees - and explains not just how they work, but why they exist and when to reach for them.

The Basic Vocabulary

A tree consists of nodes connected by edges. One node is designated the root (no parent). Nodes with no children are leaves. Every non-root node has exactly one parent. The depth of a node is the number of edges from the root to it. The height of a tree is the maximum depth of any leaf.

class TreeNode:
    def __init__(self, val: int):
        self.val = val
        self.left: "TreeNode | None" = None
        self.right: "TreeNode | None" = None

A binary tree restricts each node to at most two children, called left and right. This constraint sounds limiting but enables a crucial property: the height of a balanced binary tree with $n$ nodes is $O(\log n)$. This is why binary trees give us $O(\log n)$ operations - the height of the tree is the number of decisions you need to make to traverse from root to any leaf.

Binary Search Trees: Ordering as a Superpower

A binary search tree (BST) adds one rule to the binary tree: for every node, all values in the left subtree are smaller, and all values in the right subtree are larger. This simple invariant transforms search from $O(n)$ into $O(\log n)$.

class BST:
    def __init__(self):
        self.root: TreeNode | None = None

    def insert(self, val: int) -> None:
        if not self.root:
            self.root = TreeNode(val)
        else:
            self._insert(self.root, val)

    def _insert(self, node: TreeNode, val: int) -> None:
        if val < node.val:
            if node.left is None:
                node.left = TreeNode(val)
            else:
                self._insert(node.left, val)
        else:
            if node.right is None:
                node.right = TreeNode(val)
            else:
                self._insert(node.right, val)

    def search(self, val: int) -> bool:
        node = self.root
        while node:
            if val == node.val:
                return True
            node = node.left if val < node.val else node.right
        return False

The search is elegant: at each node, compare the target against the current value. Go left if smaller, right if larger. Each comparison eliminates half the remaining tree. This is binary search, except the structure of the tree enforces the ordering rather than requiring the array to be pre-sorted.

BST degradation: the sorted insertion problem. Here is the dark secret of BSTs. The $O(\log n)$ guarantee only holds when the tree is balanced - roughly equal numbers of nodes on left and right at every level. If you insert values in sorted order, something terrible happens:

bst = BST()
for val in [1, 2, 3, 4, 5, 6, 7]:   # sorted insertion
    bst.insert(val)

# The resulting tree looks like this:
# 1
#  \
#   2
#    \
#     3
#      \
#       4

Every node has only a right child. The tree degenerates into a linked list. Search, which was supposed to be $O(\log n)$, is now $O(n)$. Your beautiful balanced BST is now just an expensive linked list with extra steps. This is not a rare edge case - any nearly-sorted input triggers significant degradation.

Tree Traversal: Three Ways to Visit Every Node

Tree traversal is the process of visiting every node exactly once. For binary trees, three recursive orderings dominate:

def inorder(node: TreeNode | None) -> list[int]:
    if node is None:
        return []
    return inorder(node.left) + [node.val] + inorder(node.right)

def preorder(node: TreeNode | None) -> list[int]:
    if node is None:
        return []
    return [node.val] + preorder(node.left) + preorder(node.right)

def postorder(node: TreeNode | None) -> list[int]:
    if node is None:
        return []
    return postorder(node.left) + postorder(node.right) + [node.val]

Inorder (left → node → right) visits a BST in sorted ascending order. This is not a coincidence - it follows directly from the BST invariant. Preorder (node → left → right) visits a node before its subtrees, useful for serializing a tree or copying its structure. Postorder (left → right → node) visits children before parents, useful for evaluating expression trees and safe deletion (you free children before the parent, never leaving dangling pointers).

Where traversal appears in practice: compilers use postorder traversal of expression trees to generate code (evaluate operands, then apply the operator). Build systems use a form of topological sort - essentially postorder on a dependency graph - to determine build order. DOM manipulation in browsers often requires preorder traversal to process parent elements before children.

Balanced BSTs: Rotation Intuition Without the Full Proof

The solution to BST degradation is self-balancing: after each insert or delete, the tree restructures itself to maintain balance. Two famous variants are AVL trees and Red-Black trees.

Both use rotations - local rearrangements of nodes that preserve the BST ordering invariant while improving balance. A right rotation takes a left-heavy node and pivots it so its left child becomes the new parent. A left rotation does the reverse. AVL trees maintain the strict invariant that for every node, the heights of its left and right subtrees differ by at most 1. Red-Black trees use a weaker invariant (encoded in node colors) that allows slightly more imbalance but requires fewer rotations on average.

I will not walk through the full rotation rules here - they are fiddly and not illuminating on first contact. The key intuition is simpler: both approaches guarantee the tree height stays $O(\log n)$, so all operations (search, insert, delete) stay $O(\log n)$ regardless of insertion order. The cost is complexity: implementing a correct Red-Black tree from scratch is notoriously difficult, which is why you almost never should.

In Python: the standard library does not include a balanced BST. The sortedcontainers.SortedList package is the practical answer - it uses a different internal structure (a sorted list of sorted lists) but provides $O(\log n)$ operations and preserves sorted order. In production Python, you reach for that rather than implementing tree rotations.

Heaps: When You Only Care About the Minimum

A heap solves a specific problem: you have a collection of items with priorities, and you frequently need to remove the minimum (or maximum) priority item. You do not need arbitrary index access, and you do not need sorted iteration - just “give me the smallest thing” repeatedly.

A min-heap is a binary tree with two properties:

  1. Complete: every level is fully filled except possibly the last, which fills left to right.
  2. Heap property: every parent is smaller than or equal to both its children.

These two properties together mean the root is always the minimum element, and the tree height is always $O(\log n)$ (because a complete binary tree with $n$ nodes has height $\lfloor \log_2 n \rfloor$).

The Array Representation: A Stroke of Brilliance

Here is the clever insight that makes heaps so practical. Because heaps are complete binary trees, you can store them in a plain array without any pointers at all. The relationships are implicit in the indices:

  • The root is at index 1 (or 0, depending on implementation)
  • For a node at index $i$: its parent is at $\lfloor i/2 \rfloor$, its left child is at $2i$, its right child is at $2i+1$
# Heap stored as array: [_, 1, 3, 5, 7, 9, 8, 6]
# (index 0 unused; root at index 1)
#
# Node at index 1 (value 1): parent = none, left child at 2 (value 3), right child at 3 (value 5)
# Node at index 2 (value 3): parent at 1 (value 1), left child at 4 (value 7), right child at 5 (value 9)

No pointers. No node objects. Just an array and integer arithmetic. This is cache-friendly, memory-efficient, and requires no dynamic allocation. It is a beautiful example of using mathematical structure to eliminate overhead.

Python’s heapq module implements this:

import heapq

heap = []
heapq.heappush(heap, 5)
heapq.heappush(heap, 1)
heapq.heappush(heap, 3)
heapq.heappush(heap, 9)
heapq.heappush(heap, 2)

print(heap)              # [1, 2, 3, 9, 5] - array form of the heap
print(heapq.heappop(heap))   # 1 - always the minimum
print(heapq.heappop(heap))   # 2

Both heappush and heappop are $O(\log n)$ - they maintain the heap property by “bubbling up” or “bubbling down” through at most $O(\log n)$ levels.

Heapify: Building a Heap in O(n)

You might expect that building a heap from $n$ elements takes $O(n \log n)$ - just insert each element. But there is a smarter approach: start with the array in any order, then fix heap violations from the bottom up. This is $O(n)$, which is not obvious but can be proven by noting that most nodes are near the bottom and need few swaps.

data = [5, 3, 8, 1, 9, 2, 7]
heapq.heapify(data)   # O(n) - modifies in place
print(data[0])        # 1 - the minimum

Priority Queues in Practice: OS Process Scheduling

The operating system scheduler uses a priority queue (often heap-based) to decide which process gets CPU time next. Each process has a priority; the scheduler always picks the highest-priority ready process. In Linux, the Completely Fair Scheduler uses a red-black tree (ordered by “virtual runtime” - a measure of how much CPU time a process has received) rather than a heap, because it needs efficient “find minimum” and also efficient removal of arbitrary elements. The heap’s limitation is that arbitrary removal is $O(n)$; finding the minimum is $O(1)$ and removing it is $O(\log n)$, but that’s it.

Dijkstra’s shortest-path algorithm uses a min-heap as its priority queue. Each node in the graph gets a tentative distance; the heap always gives you the unvisited node with the smallest tentative distance. Without the heap, Dijkstra’s algorithm is $O(V^2)$; with a binary heap it is $O((V + E) \log V)$; with a Fibonacci heap (theoretical) it is $O(E + V \log V)$.

import heapq

class TaskScheduler:
    def __init__(self):
        self._queue: list[tuple[int, str]] = []

    def add_task(self, priority: int, name: str) -> None:
        heapq.heappush(self._queue, (priority, name))

    def run_next(self) -> str | None:
        if not self._queue:
            return None
        _, name = heapq.heappop(self._queue)
        return name

scheduler = TaskScheduler()
scheduler.add_task(3, "write report")
scheduler.add_task(1, "fix production bug")
scheduler.add_task(2, "review PR")

print(scheduler.run_next())   # "fix production bug"
print(scheduler.run_next())   # "review PR"

Heaps vs Sorted Arrays: The Right Tool for the Job

A common confusion: if you need to always have the minimum available, why not just keep a sorted array? Insert in sorted position, always access index 0.

The problem is insertion. Inserting into a sorted array at the right position is $O(\log n)$ to find the position (binary search) but $O(n)$ to shift elements. A heap gives you $O(\log n)$ for both push and pop, and the array storage means cache behavior is good.

Conversely, if you need to iterate through all elements in sorted order, a heap is the wrong tool - it only guarantees the minimum at the root, not full sorted order. For a one-time sorted traversal, sorted() is simpler. For repeatedly getting the minimum while also inserting new elements (the streaming/online case), a heap is exactly right.

Expression Trees and Parsing

When a compiler or interpreter encounters (3 + 4) * 2, it builds an expression tree internally:

    *
   / \
  +   2
 / \
3   4

Postorder traversal evaluates this correctly: visit 3, visit 4, apply +, visit 2, apply *. The tree structure naturally handles operator precedence - the shape of the tree encodes which operations are applied first, eliminating the need for explicit parentheses once the tree is built.

This is why expression trees exist as a concept in compilers: they are an intermediate representation that captures the semantics of an expression in a form that is easy to evaluate, transform, and optimize.

B-Trees: Heaps and BSTs Were Designed for RAM. Databases Need Something Else.

Here is a hard truth: BSTs, as described above, are mostly obsolete as the indexing structure inside production databases. The reason is disk I/O.

When you search a BST, each comparison involves accessing a node. In RAM, this is cheap. On disk, a random seek takes ~10ms - about 100,000 times slower than a RAM access. A balanced BST with $n$ entries has height $O(\log_2 n)$, meaning $O(\log_2 n)$ disk seeks. For a million entries, that is about 20 seeks, each at 10ms - 200ms for a single lookup. Unacceptable.

A B-tree (and its variant, the B+ tree) fixes this by making each node wider - storing many keys and many children per node. A B-tree node might hold 100 or 1000 keys, with 101 or 1001 children. The height drops to $O(\log_{1000} n)$. For a billion entries, that is just 3 levels - 3 disk seeks, 30ms, versus 30 seeks and 300ms for a BST.

Each node is sized to match a disk page (typically 4KB or 16KB), so reading a node is one disk seek regardless of how many keys it holds. This is the insight: the bottleneck is seeks, not comparisons. Make each node as wide as a disk page.

This is why every major database (PostgreSQL, MySQL/InnoDB, SQLite, SQL Server, Oracle) uses B+ trees for their primary indexes, and why BSTs, despite their elegance, are not the right tool for that job.

The Critique: When Not to Use These Structures

BSTs in Python: For most Python applications, a balanced BST is overkill. If you need sorted order and occasional insertion, sortedcontainers.SortedList works well. If you need frequent lookups, a dict is faster. If you need a sorted sequence you rarely modify, sort it once and use bisect for $O(\log n)$ binary search on a plain list - no tree required.

Heaps vs sorted(): If you need the top K elements from a list and K is small, heapq.nlargest(k, items) is the right call. If you are building an elaborate priority queue and find yourself needing arbitrary element removal or iteration in sorted order, reconsider - a different structure (like a sorted container or a set with a parallel heap) may be cleaner.

Tree traversal in production Python: Recursive tree traversal risks hitting Python’s recursion limit (default 1000) on deep trees. For production code on potentially deep trees, prefer iterative traversal with an explicit stack.

def inorder_iterative(root: TreeNode | None) -> list[int]:
    result = []
    stack = []
    current = root
    while current or stack:
        while current:
            stack.append(current)
            current = current.left
        current = stack.pop()
        result.append(current.val)
        current = current.right
    return result

Summary

Structure Search Insert Delete Best For
Binary Search Tree $O(\log n)$ avg, $O(n)$ worst $O(\log n)$ avg $O(\log n)$ avg Sorted data, not guaranteed balanced
Balanced BST (AVL/RB) $O(\log n)$ $O(\log n)$ $O(\log n)$ Sorted data with balance guarantee
Min-heap $O(n)$ arbitrary $O(\log n)$ $O(\log n)$ min only Priority queue, streaming minimum
B-tree $O(\log_M n)$ $O(\log_M n)$ $O(\log_M n)$ Disk-based indexes, databases

Trees are the right structure when your data has natural hierarchy, when you need ordered traversal alongside fast lookup, or when you need to always efficiently access the minimum. The specific variant - BST, heap, B-tree - depends on where the data lives (RAM vs disk), whether balance is guaranteed, and what operations dominate your workload.
