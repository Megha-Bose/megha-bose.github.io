Debugging and Profiling

Sep 15, 2024
13 min read

Helpful context: Writing Clean, Testable Code

On September 9, 1947, Grace Hopper’s team at Harvard found that a moth had flown into relay 70 of the Mark II computer and jammed it. They taped the moth into their logbook alongside the note “First actual case of bug being found.” The word “bug” to describe a defect had been in use since at least Thomas Edison, but the moth made it stick.

What is striking about the story is not the moth. It is the logbook. Hopper’s team had a systematic record - they documented the failure, the investigation, the cause, and the resolution. That methodical approach is what separates debugging as a craft from debugging as flailing. The moth was found because they were looking for it in the right way, with the right records.

Seventy-five years later, the gap between mental model and machine reality is still the source of almost every bug. You believe the code does one thing. The code does something else. Debugging is the process of discovering the discrepancy and collapsing it.

The History of Looking at Running Programs

For most of computing’s early history, debugging meant staring at printouts. Programs ran in batch mode - you submitted your punch cards, the machine ran, and you got output. If it was wrong, you read the printout and thought.

Printf debugging is the oldest interactive debugging technique: insert print statements, run the program, read the output, repeat. It works. Grace Hopper’s generation used it. Programmers in 2024 still use it because sometimes it is the fastest path to the answer.

Interactive debuggers arrived in the 1960s. DDT (Dynamic Debugging Technique) for the DEC PDP systems let programmers pause execution, inspect registers, and step through instructions. By the 1980s, gdb had become the standard for C programming - attach to a running process, set breakpoints, inspect memory. The concept has not changed in forty years; the interface has become significantly more comfortable.

IDE debuggers in the 1990s and 2000s put a graphical face on the same operations. Click a line number to set a breakpoint. Hover over a variable to see its value. Step through code with arrow keys. VS Code and PyCharm do this today; under the hood, they are driving the same mechanisms as gdb.

Time-travel debugging is newer and remarkable. Tools like Mozilla’s rr and Visual Studio’s IntelliTrace record a complete execution history and allow you to run the program backwards - step backwards through code, inspect the state at any past moment. This is transformative for bugs that are hard to reproduce, or bugs where you know what the final wrong state is but need to find where it came from. rr is currently Linux-only; it works by recording all non-determinism (thread interleaving, system calls) and replaying it deterministically.

The Debugging Mindset: The Scientific Method in Code

Good debugging is not intuition, it is methodology. The structure that works:

1. Reproduce it. A bug you cannot reproduce consistently is a bug you cannot fix reliably. Before anything else, find the minimal input that triggers the problem every time. Flaky bugs that only sometimes appear are usually caused by race conditions, external state, or timing - which is important information.

2. Observe precisely. What exactly happens versus what do you expect to happen? “It’s broken” is not useful. “The function returns -1 when given an empty list, but I expected it to return 0” is a precise, actionable observation.

3. Hypothesize. Form a specific, testable hypothesis about the cause. Not “something is wrong with the database” but “the database query returns null when the user has no orders, and the code does not handle that case.”

4. Test the hypothesis. Design the smallest possible intervention that would confirm or refute it. Add a print statement, run a specific test case, check a variable’s value at a specific point. Do not make multiple changes at once - you will not know which one fixed it.

5. Update and repeat. If your hypothesis was wrong, you have learned something. The search space has narrowed. Form a new hypothesis. This is not failure - this is science.

The most common debugging mistake is jumping to step 4 (trying a fix) before completing step 3 (understanding the cause). A guess that happens to work leaves you not knowing why, which means the same bug, or a cousin of it, comes back. A fix that follows from understanding is durable.

Reading Tracebacks: Bottom Up, Not Top Down

Python’s tracebacks feel overwhelming at first glance. The instinct is to read from the top. Wrong - read from the bottom.

The last line tells you what happened: the exception type and its message. The lines above show the call chain that led there. The frame immediately above the bottom is almost always where your code did something wrong.

Traceback (most recent call last):
  File "main.py", line 10, in <module>
    result = process(data)
  File "main.py", line 5, in process
    return data["key"] * 2
KeyError: 'key'

Bottom up: KeyError: 'key' - a key access failed. The line above: data["key"] in the function process. That is your culprit. The question is then: why doesn’t data have "key"? Is data the wrong object? Did someone forget to set it? Was it deleted? The traceback tells you where; understanding the data flow tells you why.

Common Python exceptions and what they actually mean:

  • KeyError: dict access with a missing key. Usually means the data arriving at that point doesn’t have the shape you expected.
  • AttributeError: 'NoneType' object has no attribute 'X': you called .X on None. A function somewhere returned None when you expected an object. Hunt the function that should have returned something.
  • TypeError: unsupported operand type(s): you applied an operation to the wrong type. Often str where int was expected, or None where a number was expected.
  • IndexError: list access out of bounds. An off-by-one error, or processing a shorter list than assumed.
  • RecursionError: your recursive function has no base case, or the input is large enough to exhaust the call stack.

The Python Debugger: pdb

print() debugging is fine. But when you are doing multiple rounds of print-run-read-repeat, a proper debugger is dramatically faster. Python ships with pdb.

The entry point since Python 3.7:

def process(items):
    total = 0
    for item in items:
        breakpoint()    # execution pauses here; drops you into pdb
        total += item
    return total

When execution hits breakpoint(), you get an interactive prompt. Key commands:

Command Action
n Next line (steps over function calls)
s Step into a function call
c Continue to the next breakpoint
p expr Print the value of an expression
pp expr Pretty-print (useful for dicts and lists)
l List source code around current line
w Where am I - show the call stack
b 42 Set a breakpoint at line 42
q Quit the debugger

The power of the debugger over print statements is interactivity: you can inspect arbitrary expressions, navigate up and down the call stack, set breakpoints conditionally, and modify variables at runtime to test hypotheses without restarting the program.

VS Code and PyCharm wrap pdb in a graphical interface. The underlying mechanism is identical - they are calling the same Python debug API. If you understand pdb, you understand what the GUI is doing.

When to use print debugging vs a proper debugger: print debugging is faster for simple “what value does X have here” questions. The debugger shines for complex control flow - loops with many iterations, code paths that branch in non-obvious ways, or bugs that involve the interaction of several functions. If you have added more than three print statements and still have not found the bug, open the debugger.

Binary Search on Hypotheses

When you have no idea where a bug is, the fastest approach is binary elimination. Does the bug occur at point A? Does it occur at point B (halfway through the code)? If yes, it is in the second half; now bisect that half. If no, it is in the first half.

This is literally the git bisect command for commit-level bugs: you tell git “this commit is good, this commit is bad,” and it binary-searches through the commit history to find the exact commit that introduced the regression. For any codebase with a test that reliably detects the bug, git bisect run pytest test_specific.py will find the culprit commit automatically.

git bisect start
git bisect bad HEAD           # current commit is broken
git bisect good v1.2.0        # this release was fine
git bisect run pytest tests/test_regression.py   # automated search

The same bisection instinct applies within a single function: add an assertion in the middle. If it fails, the bug is before that point. If it passes, the bug is after. Bisect until you have isolated a single expression.

Rubber Duck Debugging: Why It Actually Works

There is a technique - often laughed at, consistently effective - called rubber duck debugging. You explain your code, line by line, to a rubber duck (or a colleague, or a houseplant). The explanation process itself frequently reveals the bug.

This is not magic, and the duck is not helping. What happens is that explaining code forces you to externalize your mental model and compare it, step by step, to what the code actually does. Most bugs exist because your mental model drifts from reality. When you read your own code silently, your brain fills in gaps with what you believe should be there. When you narrate it explicitly - “and then I take the index and add one, because…” - you are forced to confront whether that “because” is actually true.

Practically: if you have been staring at a bug for more than 30 minutes, stop and explain the relevant code to someone else (or a rubber duck, or write it out in a comment). The act of narrating often short-circuits the mental model that is hiding the bug.

Common Bug Patterns

Off-by-one errors. The most frequent bug in code that handles sequences. Usually involves < vs <=, zero-indexing confusion, or loop bounds. When something works for n elements but fails for n+1, or works in general but fails on the first or last element, look for an off-by-one.

# Bug: should be range(len(arr) - 1)
for i in range(len(arr)):
    if arr[i] > arr[i + 1]:   # IndexError on last iteration
        arr[i], arr[i + 1] = arr[i + 1], arr[i]

Mutable default arguments. Python evaluates default arguments once, at function definition time, not each call.

# Bug: all calls share the same list
def append_to(item, result=[]):
    result.append(item)
    return result

append_to(1)    # [1]
append_to(2)    # [1, 2] - the list persisted!

# Fix: use None as sentinel
def append_to(item, result=None):
    if result is None:
        result = []
    result.append(item)
    return result

Reference vs copy. Assigning a list does not copy it.

a = [1, 2, 3]
b = a          # b is a reference to the same list
b.append(4)
print(a)       # [1, 2, 3, 4] - a was modified!

b = a[:]       # shallow copy
b = a.copy()   # also shallow copy
import copy; b = copy.deepcopy(a)   # deep copy (nested structures)

Mutation during iteration. Modifying a list while iterating over it produces undefined behavior - skipped elements, exceptions, or infinite loops.

# Bug: modifying the list you're iterating
items = [1, 2, 3, 4, 5]
for item in items:
    if item % 2 == 0:
        items.remove(item)   # skips elements!

# Fix: iterate over a copy
for item in items[:]:
    if item % 2 == 0:
        items.remove(item)

# Or, more Pythonically:
items = [item for item in items if item % 2 != 0]

Profiling: Measure Before You Optimize

“Premature optimization is the root of all evil.” Knuth wrote that in 1974, and programmers still need to hear it. The reason is not that optimization is bad - it is that optimization applied to the wrong part of the code is wasted effort at best and a source of new bugs at worst.

The 80/20 rule applies to performance: 80% of execution time is typically spent in 20% of the code. Your job is to find that 20% and optimize only there. Without measurement, you are guessing, and programmers are reliably bad at guessing where their code spends time.

cProfile is Python’s built-in profiler. It instruments every function call and reports cumulative time, per-call time, and call counts:

import cProfile
import pstats

def slow_function(n):
    return sum(i * i for i in range(n))

with cProfile.Profile() as pr:
    slow_function(1_000_000)

stats = pstats.Stats(pr)
stats.sort_stats("cumulative")
stats.print_stats(10)   # top 10 by cumulative time

Or without modifying your code:

python -m cProfile -s cumtime my_script.py

The output tells you which function consumed the most total time. Start there.

line_profiler goes one level deeper: it shows time per individual line within a function. Install with pip install line-profiler. Decorate the function you care about with @profile and run kernprof -l -v script.py.

@profile   # only needed when running with kernprof
def process_data(items):
    result = []
    for item in items:
        result.append(item * 2)   # this line: X% of time
    return result

memory_profiler tracks memory usage per line - useful when you have a memory leak or unexpectedly high memory consumption. pip install memory-profiler, same @profile decorator, run with python -m memory_profiler script.py.

The workflow: profile first with cProfile to find the slow function. Profile that function with line_profiler to find the slow line. Optimize only that line. Re-profile to confirm improvement.

Observability in Production

The techniques above work in development where you can run the code interactively. Production systems are different: you cannot attach a debugger to a live server serving real traffic. Instead, you need the system to tell you what is happening through logs, metrics, and traces.

Structured logging is print-debugging at scale. Instead of free-form strings, log machine-readable JSON with consistent fields. This makes logs searchable and aggregatable across millions of requests.

import logging
import json

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)

def process_order(order_id: str, user_id: str):
    logger.info(json.dumps({
        "event": "order_processing_started",
        "order_id": order_id,
        "user_id": user_id,
    }))
    # ... processing ...
    logger.info(json.dumps({
        "event": "order_processing_complete",
        "order_id": order_id,
        "duration_ms": 42,
    }))

Distributed tracing (OpenTelemetry, Jaeger, Zipkin) extends this to systems composed of many microservices. A single user request might touch ten services; distributed tracing assigns a trace ID to the request and propagates it, so you can reconstruct the full path a request took and see where time was spent or where errors occurred. This is the production equivalent of a call stack trace.

The critique of over-relying on the debugger: pausing a debugger is a manual, one-request-at-a-time operation. A production bug that only occurs 0.1% of the time cannot be caught with a debugger - it requires instrumentation built into the system. Systems that are observable by design (with good logging, metrics, and tracing) are easier to debug in production than systems that are only debuggable in development with an attached debugger. The best debugging happens before deployment, through tests; the second best is observability; the debugger is the last resort.

Summary

Technique Best For Limitations
print() debugging Quick, simple “what value is here?” No persistence, hard to disable, verbose at scale
pdb / breakpoint Complex control flow, interactive exploration Requires running locally, one request at a time
cProfile Finding the slow function Function granularity only
line_profiler Finding the slow line within a function Requires installing extra package
Structured logging Production debugging, audit trails Must be designed in advance
Distributed tracing Debugging across multiple services Operational overhead to set up
git bisect Finding which commit introduced a regression Requires a reliable reproducing test
Time-travel debugging (rr) Bugs that are hard to reproduce Linux only, high recording overhead

Debugging is a skill that sharpens through practice. The tools are learnable in an afternoon. The discipline - form hypotheses, test them precisely, measure before optimizing, build systems that are observable by design - takes longer and pays dividends throughout a career.
