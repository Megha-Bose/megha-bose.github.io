Data Compression

Jun 1, 2024
9 min read

Shannon’s source coding theorem gives you the floor: no lossless code can compress a source $X$ below $H(X)$ bits per symbol on average. But knowing the floor exists is different from knowing how to build a ladder that reaches it. In 1948, the proof was existential - there are codes that get close, but the construction was not practical. The history of data compression since then is the story of building practical codes that approach, and in some cases effectively reach, Shannon’s limit.

The story splits early. Huffman coding, invented in 1952, gets within 1 bit per symbol of entropy - not bad, but that overhead is significant for short messages and small alphabets. Arithmetic coding, developed in the 1970s and 1980s, eliminates most of that gap by coding entire messages rather than individual symbols. Lempel and Ziv showed in 1977 and 1978 that you do not even need to know the source distribution - you can learn it on the fly and still converge to the entropy rate. These three threads - optimal symbol codes, range coding, and universal coding - are the foundations of every modern compression format.

The second half of the story is lossy compression, where you are willing to accept some distortion in exchange for much higher compression ratios. JPEG images, MP3 audio, and H.265 video all discard information you were not going to notice anyway. The theory behind this trade-off - when you can discard, and how much - is rate-distortion theory. But understanding lossless compression is the prerequisite.

Why Huffman Leaves a Gap

Huffman coding assigns each source symbol an integer-length codeword. If $p(x)$ is the probability of symbol $x$, the ideal codeword length is $-\log_2 p(x)$ bits. But codeword lengths must be integers, so the actual length is $\lceil -\log_2 p(x) \rceil$, which rounds up by at most 1 bit.

For a single symbol, the expected length satisfies

$$H(X) \leq L_{\text{Huffman}} < H(X) + 1.$$

The upper bound of $H(X) + 1$ is tight. Consider a source with two symbols: $p(a) = 0.01$, $p(b) = 0.99$. Entropy is $H \approx 0.081$ bits, but any binary Huffman code assigns at least 1 bit per symbol - nearly a factor of 12 overhead.

The fix is to code blocks. If you encode $n$ symbols at a time, treating sequences of $n$ symbols as a single “super-symbol,” the overhead per symbol drops to less than $1/n$ bits. In the limit $n \to \infty$, Huffman codes on blocks approach $H(X)$. But the alphabet of $n$-symbol blocks has size $|\mathcal{X}|^n$, so the table size explodes. Arithmetic coding avoids this by working directly on infinite-length sequences.

Arithmetic Coding: Approaching the Limit

Arithmetic coding encodes an entire message as a single number in $[0, 1)$. The algorithm maintains an interval $[l, r)$ that narrows as symbols are processed. For each new symbol $x$ with probability $p(x)$, the interval is split proportionally, and the sub-interval corresponding to $x$ is retained.

After processing a sequence $x_1, x_2, \ldots, x_n$ with probability $P(x_1, \ldots, x_n)$, the interval has width $P(x_1, \ldots, x_n)$. Any number in this interval serves as a codeword; it takes $\lceil -\log_2 P(x_1, \ldots, x_n) \rceil + 1$ bits to specify it. For a sequence of $n$ i.i.d. symbols, this is approximately $\sum_i -\log_2 p(x_i) + 1$ bits total - getting within 1 bit of the optimal for the entire message, not per symbol.

The expected length per symbol is

$$H(X) \leq \frac{L_{\text{arithmetic}}}{n} < H(X) + \frac{1}{n}.$$

As $n \to \infty$, this overhead vanishes. Arithmetic coding effectively achieves $H(X)$ bits per symbol for long messages. It is the practical standard when you want to get close to Shannon’s limit: JPEG2000, H.265 (HEVC), and CABAC in H.264 all use arithmetic coding internally.

Implementation note. Multiplying probabilities causes underflow quickly. Real implementations use integer arithmetic with renormalization (range coding), updating the interval with integer arithmetic and emitting bits as the interval is determined. The result is efficient enough for hardware video encoders.

Prefix-Free Codes and the Kraft Inequality

Both Huffman and arithmetic coding produce prefix-free codes: no codeword is a prefix of another. This property is what allows unambiguous decoding: you can read bits from a stream and know exactly when a codeword ends without any delimiter.

The fundamental constraint on prefix-free codes is the Kraft inequality:

$$\sum_{x} 2^{-\ell(x)} \leq 1,$$

where $\ell(x)$ is the codeword length for symbol $x$. Conversely, any set of lengths satisfying this inequality can be realized by a prefix-free code. The inequality is tight for Huffman codes (equality holds).

The Kraft inequality connects code design to probability: assign probability $2^{-\ell(x)}$ to each codeword, and the sum is at most 1. Optimal code design is equivalent to optimal probability assignment, which is why entropy (the expected value of $-\log p(x)$) bounds the expected code length from below.

LZ77 and LZ78: Universal Compression

Huffman and arithmetic coding require knowledge of the source probabilities. Lempel and Ziv showed in 1977 and 1978 that you can compress without knowing the distribution, and still achieve the entropy rate asymptotically.

LZ77 (used in gzip, deflate, PNG, HTTP compression): maintains a sliding window of recently seen data. When encoding the next piece of data, search the window for the longest match. Output a triple (offset, length, next character): “copy $\ell$ characters starting from position $-d$ in the window, then output character $c$.” If the source has any statistical regularity - repeated words in text, repeated patterns in code - the matches will be long and the output will be short.

LZ78 and its variant LZW (used in GIF, early Unix compress): builds a dictionary incrementally. Start with single-character entries. When encoding, find the longest prefix that is already in the dictionary, output its index, then add the current match extended by one character as a new entry. LZW omits the new character from the output (the decoder can infer it), reducing overhead further.

Why they work. Lempel and Ziv proved that both algorithms achieve the entropy rate $H$ of any stationary ergodic source as the message length $n \to \infty$. The dictionary learns the source statistics on the fly - the compression ratio for an English text file will converge to the entropy rate of English, estimated around 1 to 1.5 bits per character, even though the algorithm never sees a probability table.

Modern descendant: Zstandard (zstd). Developed at Facebook (now Meta), zstd combines LZ77-style dictionary matching with finite state entropy (FSE), a fast arithmetic coder based on asymmetric numeral systems (Jarek Duda, 2013). It achieves compression ratios close to gzip with 5-10 times faster decompression. Used by the Linux kernel, web servers, database engines, and cloud storage systems.

JPEG: Lossy Compression in Practice

JPEG compression illustrates the principle of exploiting perceptual irrelevance:

  1. Color space conversion. Convert RGB to YCbCr (luminance and chrominance). Human eyes are more sensitive to luminance detail than color detail, so chrominance channels are downsampled.

  2. Block DCT. Divide the image into $8 \times 8$ pixel blocks. Apply the discrete cosine transform to each block. The result is 64 frequency coefficients per block, from low-frequency (large-scale structure) to high-frequency (fine detail).

  3. Quantization. Divide each frequency coefficient by a quantization step size (larger for high frequencies) and round to the nearest integer. This step is lossy - it throws away high-frequency detail that the eye barely perceives. The quality parameter in JPEG controls these quantization step sizes.

  4. Entropy coding. The quantized coefficients are run-length encoded (exploiting the many zeros in high-frequency components) and Huffman-coded.

The quality - distortion tradeoff is entirely in step 3. A larger quantization step means more coefficients round to zero, fewer bits are needed, and more detail is lost. The optimal quantization step for a given bit budget is what rate-distortion theory analyzes formally.

H.265/HEVC: Exploiting Temporal Redundancy

Video compression extends the JPEG idea to sequences of frames. Most of a video frame is nearly identical to the previous frame (the background does not move), and this temporal redundancy is the main opportunity.

H.265 (HEVC) uses four stages:

  1. Prediction. For each block, find a similar block in a previous (or future) reference frame (inter prediction) or predict from neighboring blocks in the same frame (intra prediction). Output the prediction error (residual).
  2. Transform. Apply a DCT-like transform to the residual. The residual is small (since prediction is good), so most coefficients will be small or zero.
  3. Quantization. Same principle as JPEG - round to reduce bit cost.
  4. Entropy coding (CABAC). Context-adaptive binary arithmetic coding, adapting the probability model to local statistics.

H.265 achieves roughly the same visual quality as H.264 at half the bit rate. This improvement came from larger and more flexible block sizes (up to $64 \times 64$ vs $16 \times 16$), better motion compensation, and improved entropy coding. The mathematical structure - predict, transform, quantize, entropy code - is the same as JPEG.

The Incompressibility Argument

A compression algorithm maps $n$-bit strings to shorter strings. But there are $2^n$ strings of length $n$ and only $\sum_{k=0}^{n-1} 2^k = 2^n - 1$ strings of length less than $n$. By pigeonhole, at least one $n$-bit string cannot be compressed to fewer than $n$ bits. In fact, almost all strings cannot be compressed:

Claim. At most $2^{n-k}$ strings of length $n$ can be compressed to fewer than $n - k + 1$ bits.

Proof. There are only $2^{n-k}$ binary strings of length less than $n - k + 1$. $\square$

So a uniformly random $n$-bit string can be compressed by more than $k$ bits with probability at most $2^{-k}$. Most bit strings are incompressible. The strings that do compress are exactly those with low entropy - statistical regularities that the compressor can exploit.

Practical corollary. Do not compress already-compressed data. ZIP files, JPEGs, and encrypted data are approximately uniformly random. Running gzip on a JPEG file will produce a file the same size or larger, because there is no redundancy left to exploit. Compression exploits structure; random-looking data has none.

Summary

Method Type Achieves Used in
Huffman coding Lossless, symbol code $H(X)$ to $H(X)+1$ bits/symbol HTTP, older formats
Arithmetic coding Lossless, sequence $H(X)$ + $O(1/n)$ bits/symbol JPEG2000, HEVC/CABAC
LZ77 Lossless, universal Entropy rate (no prior needed) gzip, deflate, PNG, zstd
LZ78 / LZW Lossless, universal Entropy rate GIF, early compress
JPEG Lossy Determined by quantization step Images everywhere
H.265/HEVC Lossy Determined by quantization + motion Video streaming

Shannon’s source coding theorem sets the floor. Huffman gets close for symbols. Arithmetic coding and LZ methods get essentially all the way there for sequences. The gap between theoretical entropy and practical compression ratio has shrunk to near zero for lossless formats; for lossy compression, the remaining question is how much distortion you are willing to accept.
