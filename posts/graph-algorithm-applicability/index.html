Graph Algorithm Applicability - Choosing the Right Tool for the Graph

Aug 3, 2021
5 min read

Prerequisites:

  • Graph Algorithms & Traversal - Maps, Networks, and Hidden Order
  • Minimum Spanning Trees & Union-Find - Spanning a World on a Budget
  • Shortest Paths (Dijkstra, Bellman-Ford)
  • Network Flow & Matching

By now you have seen a lot of graph algorithms. The question that naturally arises: for a given problem and a given graph, which algorithm can you actually use? The answer depends on the structure of the graph - directed or undirected, weighted or unweighted, DAG or not, negative weights or not. Getting this wrong means applying an algorithm where its assumptions are violated, producing incorrect results or running forever.

This post is a reference. Come back to it whenever you are looking at a graph problem and need to reason about which tools apply.

The Graph Properties That Matter

Before picking an algorithm, pin down four properties of your graph:

1. Directed or undirected? In an undirected graph, every edge goes both ways. In a directed graph (digraph), edges are one-way. Many algorithms are defined for one but not the other.

2. Weighted or unweighted? Weights on edges represent cost, distance, or capacity. Algorithms designed for weighted graphs can be run on unweighted graphs (set all weights to 1), but not the other way around.

3. Does it contain cycles? A directed graph with no cycles is a DAG. Several algorithms only work correctly on DAGs - they rely on processing vertices in topological order, which only exists when there are no cycles.

4. Are any edge weights negative? Negative weights are legal in many settings (profit vs. cost, altitude changes) but they break algorithms that assume once a vertex is settled it cannot be improved.

Why Each Constraint Exists

Why do topological sort and DAG-DP require a DAG?

Both algorithms process vertices in dependency order - each vertex is handled only after all its predecessors. That order only exists when the graph has no cycles. If vertex A depends on B and B depends on A, there is no valid starting point. DFS detects cycles precisely so you can check whether topological sort is applicable before running it.

Why does Dijkstra’s fail on negative weights?

Dijkstra’s greedy step finalises a vertex the moment it is extracted from the priority queue, assuming no shorter path to it will be discovered later. A negative-weight edge can always produce a shorter path through a vertex that was already finalised, violating that assumption. Consider:

A --1--> B --(-3)--> C
A -------2--------> C

Dijkstra finalises C via A→C (cost 2) before it processes B. But A→B→C costs 1 + (-3) = -2, which is cheaper. The finalised answer is wrong. Bellman-Ford avoids this by relaxing every edge $V-1$ times, without any greedy finalisation, so it can correct earlier answers.

Why do MST algorithms require undirected graphs?

A spanning tree is inherently undirected - it connects all vertices using edges that can be traversed either way. In a directed graph, the analogous problem is a minimum spanning arborescence (a directed tree rooted at one vertex, with all edges pointing away from the root). This requires Edmonds' algorithm (also called Chu-Liu/Edmonds'), which is more complex. Kruskal’s and Prim’s assume you can use each edge in either direction.

Why does Bellman-Ford require no negative cycles?

A negative cycle is a cycle whose total edge weight is negative. If a shortest path passes through a negative cycle, you can always reduce the path cost by going around the cycle one more time - there is no finite minimum. Bellman-Ford detects this: after $V-1$ relaxation rounds, if any distance can still be reduced, a negative cycle exists. The shortest path problem is simply undefined when negative cycles are reachable from the source.

Why does max flow require a directed graph?

Flow has a direction - it moves from source to sink. An undirected edge between $u$ and $v$ would mean flow could go either way simultaneously, which needs to be modelled as two directed edges with shared capacity. Network flow algorithms are always stated for directed graphs; undirected flow problems are converted by replacing each undirected edge with two directed edges.

Decision Guide

When you face a graph problem, work through these questions in order:

Summary

The key insight is that graph algorithm constraints are not arbitrary - each one reflects a genuine mathematical assumption the algorithm relies on. Understanding why a constraint exists tells you both when to apply the algorithm and what will go wrong if you don’t.

If your graph has… Watch out for…
Directed edges MST algorithms (need undirected); use arborescence algorithms instead
Cycles Topological sort and DAG-DP (undefined); check with DFS first
Negative edge weights Dijkstra’s (wrong answers); switch to Bellman-Ford
Negative cycles Bellman-Ford and Floyd-Warshall (undefined shortest paths)
Undirected edges SCCs and max flow (directed concepts); re-model if needed
