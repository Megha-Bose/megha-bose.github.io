Load Balancing & Proxies

Sep 15, 2024
10 min read

Prerequisite: Latency, Throughput & Queues · Networking: IP, TCP, HTTP

Ten servers. One million requests per second. How do you distribute work fairly - and what happens when three of those servers die at 2am?

The naive answer is “a load balancer,” but that answer is a starting point, not a conclusion. The real question is: which algorithm distributes work fairly for your workload, what happens when the load balancer itself fails, and how does the statefulness of your application change every assumption in the design?

This is a story that evolved over about thirty years, from a DNS hack to globally-aware anycast routing. The evolution is worth tracing because each step was a response to a concrete failure mode of the previous approach.

A Brief History of Getting Work to the Right Machine

In the early web era, “load balancing” meant DNS round-robin: the same domain resolved to multiple IP addresses in rotation. Browser asks for example.com, gets 10.0.0.1. Next browser asks, gets 10.0.0.2. Beautifully simple. Also deeply broken: DNS responses are cached, so a user might keep hitting a dead server for minutes after its A record should have been rotated out. And DNS knows nothing about server health.

The 2000s brought hardware load balancers - appliances from F5 Networks and Cisco, purpose-built for high-throughput packet routing. These worked well for large enterprises that could afford six-figure hardware and had dedicated networking staff to configure them. But they were rigid, expensive, and slow to reconfigure as application topology changed.

Then came software load balancers: HAProxy (2001) and Nginx (2004) brought L7 load balancing to commodity hardware. Engineers could now define backend pools in a config file, hot-reload configuration, and run the same software on $50/month cloud VMs. The F5 market didn’t disappear, but it stopped being the only option.

Finally: cloud-native load balancers - AWS ELB (2009), later ALB and NLB, GCP Cloud Load Balancing, Azure Load Balancer. These are managed services where the load balancer itself is infinitely scaled, globally distributed, and integrated with auto-scaling groups, health checks, and SSL certificate management. The engineering team no longer owns load balancer infrastructure; they configure it.

Layer 4 vs Layer 7: What the Load Balancer Can See

The layer at which a load balancer operates determines what it knows about the traffic - and what routing decisions it can make.

Layer 4 (transport layer) load balancers route at the TCP/UDP level. They see source IP, destination IP, and port. They know nothing about HTTP paths, cookies, or request bodies. This ignorance is also their strength: because they don’t parse application-layer content, they’re extremely fast - capable of handling millions of connections with microsecond-level overhead. AWS Network Load Balancer (NLB) operates at this layer and can sustain millions of requests per second with ultra-low latency.

Layer 7 (application layer) load balancers parse HTTP, HTTP/2, and gRPC. They can route based on URL path, HTTP method, headers, cookies, or query parameters. This enables patterns impossible at L4: route /api/* to the API fleet and /static/* to a CDN-backed pool; route requests with X-Beta: true to a canary deployment; terminate TLS and forward plaintext to backends. AWS Application Load Balancer (ALB) operates here, supporting content-based routing, WebSocket, and HTTP/2 natively.

The price of L7 intelligence is CPU overhead for parsing and slightly higher latency per request. Most production systems use L7 at the edge (ALB for external traffic) and L4 for internal east-west routing where path-based decisions aren’t needed and raw throughput matters.

Balancing Algorithms: More Than Round-Robin

Round-robin cycles through backends sequentially. Simple, effective when requests have similar cost and servers have similar capacity. Falls apart when some requests are 1ms and others are 5 seconds - the unlucky backend accumulates slow requests while the lucky one sits idle.

Weighted round-robin assigns a weight to each backend. A server with 8 cores gets twice the requests of a server with 4 cores. Useful during rolling upgrades when new and old instance types temporarily coexist in the same pool.

Least connections routes each new request to the backend with the fewest active connections. Far superior to round-robin for variable-duration requests. The overhead: the load balancer must maintain a global connection count, which is a coordination cost at very high connection rates.

Consistent hashing maps requests to backends using a hash of a request attribute - typically user ID or session key. The same user always hits the same backend. Critical for stateful workloads: distributed caches where eviction behavior depends on which node holds the key, or session-bearing services. The careful design: use a virtual node ring to ensure even distribution even when backends are added or removed. When a node fails, only its fraction of the keyspace needs to be remapped rather than reshuffling everything.

Power of two choices picks two backends at random and routes to the whichever has fewer connections. This achieves near-optimal load distribution with far less overhead than tracking all backends - a elegant result from probabilistic load balancing theory. Nginx Plus and Envoy support this.

How AWS ALB Actually Works

AWS ALB is worth understanding in detail because it’s the load balancer most teams encounter first and because its design reveals what cloud-native load balancing looks like at scale.

An ALB sits in front of target groups - logical pools of instances, Lambda functions, or IP addresses. Routing rules inspect incoming requests and forward to the matching target group. Rules can match on host header (for multi-tenant virtual hosting), URL path, query string parameters, HTTP method, or source IP CIDR. This makes ALB the natural place to implement blue-green deployments (two target groups, one weighted routing rule) and canary releases (5% of traffic to the canary target group, 95% to production).

ALB terminates SSL - your backends receive plaintext HTTP. This moves certificate management out of your application and into AWS Certificate Manager, where renewal is automatic and private keys never touch your application instances. Backends can optionally receive re-encrypted HTTPS (end-to-end encryption) when compliance requires it.

ALB performs health checks against each target on a configurable path, port, and interval. Unhealthy targets are automatically drained: in-flight requests complete (up to the deregistration delay, default 300 seconds) before the target is removed. This prevents hard disconnections during rolling deploys.

ALB itself is not a single machine - it scales automatically and is designed with no single point of failure. The load balancer is load-balanced. AWS SNS can route alerts when unhealthy target counts cross thresholds, giving you visibility into the health of your fleet from the load balancer’s perspective.

The Sticky Session Problem

Sticky sessions - also called session affinity - route requests from the same client to the same backend. Cookie-based affinity is the most reliable form: the load balancer injects a cookie (AWS ALB uses AWSALB) on the first response; subsequent requests carry that cookie, and the LB routes accordingly.

Sticky sessions solve a real problem: if a user’s session state lives in memory on backend-3, routing their next request to backend-5 breaks their session. But the solution creates a larger problem.

When backend-3 fails, every user pinned to it loses their session simultaneously. You haven’t eliminated the failure - you’ve concentrated it. Worse, sticky sessions defeat load balancing: if one user’s session is computationally expensive, that backend accumulates load while others sit idle. Horizontal scaling becomes less effective because traffic is pinned rather than distributed.

The stateless design principle exists to eliminate this problem entirely. If session state lives in Redis or a database rather than server memory, any backend can serve any request. The load balancer can route round-robin, least-connections, or however it likes - there’s no state to lose when a backend fails. This is not just an academic preference; it’s the prerequisite for true horizontal scaling. Sticky sessions are the architectural smell that tells you session state has leaked into your compute layer.

Health Checks: Active and Passive

A load balancer that routes to unhealthy backends is worse than no load balancer at all - it concentrates failures rather than routing around them.

Active health checks send periodic probes to each backend - typically an HTTP GET to a /health or /ready endpoint - and mark backends unhealthy if they fail. AWS ALB performs active health checks on a configurable interval (default 30 seconds) with a configurable threshold (default 3 failures before marking unhealthy). The /health endpoint should check not just that the process is running but that it can reach its dependencies: database connections open, Redis reachable, downstream services responding.

Passive health checks observe real traffic. If a backend returns 5xx responses above a threshold or consistently times out, it is temporarily removed without any probe traffic. This catches failure modes that health check endpoints don’t surface - a backend that returns 200 on /health but 503 on actual requests because its database pool is exhausted.

Most production systems use both. The subtle distinction: active checks catch dead backends quickly; passive checks catch degraded-but-alive backends that active probes miss.

Global Load Balancing: Routing Across Regions

Once your system spans multiple cloud regions, load balancing becomes a routing problem at the DNS and BGP level.

GeoDNS (AWS Route 53 with latency-based routing or geolocation policies) returns different IP addresses based on where the DNS query originates. A user in Tokyo resolves api.example.com to your ap-northeast-1 deployment; a user in Frankfurt resolves to eu-west-1. Route 53’s health checks integrate with latency-based routing: if ap-northeast-1 becomes unhealthy, Route 53 automatically routes Tokyo users to the next-closest region. TTL values matter here - a short TTL (60s) means faster failover at the cost of more DNS queries; a long TTL means cheaper operation but slower response to regional failures.

Anycast assigns the same IP address to servers in multiple datacenters. BGP routing naturally delivers packets to the topologically closest datacenter. GCP Cloud Load Balancing uses anycast globally - a single IP serves users from the nearest Google edge PoP in over 100 locations. Failover is automatic and fast (BGP convergence, typically seconds). AWS Global Accelerator uses anycast to route traffic to the nearest AWS edge location and then to the nearest healthy regional endpoint over AWS’s backbone network, avoiding the public internet for the long-haul leg.

Global Server Load Balancing (GSLB) combines health monitoring with DNS-based routing to implement disaster recovery. Route 53 with health checks and weighted routing can implement an active-passive failover: primary region gets 100% of traffic when healthy; on health check failure, Route 53 automatically shifts traffic to the secondary region.

The Load Balancer as a Single Point of Failure

Here is the irony: load balancers exist to eliminate single points of failure, but a single load balancer is itself a single point of failure.

The traditional solution: run load balancers in an active-passive HA pair. Both run the same configuration; a virtual IP floats between them via Keepalived/VRRP. When the primary fails, the secondary takes the VIP within seconds. HAProxy is commonly deployed this way.

Cloud-native load balancers (AWS ALB, GCP Cloud LB) handle this transparently - the service is distributed across multiple availability zones and scales automatically. The trade-off is control: you cannot inspect or modify the underlying load balancer infrastructure.

Envoy and service meshes (Istio, Linkerd) take a different approach: distribute the load balancing function across every service instance as a sidecar proxy, rather than concentrating it in a centralized LB. Each Envoy sidecar handles load balancing, health checking, retries, and circuit breaking for its parent service’s outbound connections. The control plane (Istio’s istiod) distributes routing configuration to all sidecars. The result: no single load balancing tier to fail. The cost: significant operational complexity and the latency overhead of two additional proxy hops for every inter-service call.
